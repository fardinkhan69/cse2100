# Clinic OS admin frontend with mock data layer

A nine-page admin panel (dashboard, patients, appointments, billing, staff, settings, notifications) was added behind a shared `DashboardLayout` shell. All data comes from an in-memory mock service (`mockData.ts`) exporting typed arrays and objects. The layout provides a collapsible sidebar, mobile Sheet overlay, notification badge, and user dropdown using existing Shadcn UI primitives. Routes are wired under `/admin/*` and wrapped in `PrivateRoute`.

**Watch for:** Admin routes use `PrivateRoute` (authentication-only) rather than `AdminRoute` (role-checked), meaning any logged-in user can access billing, staff, and settings pages (**confirmed**). All page state is ephemeral — form submissions, mark-as-read, settings saves all fire toasts but mutate only local component state, so navigating away and back resets everything (**confirmed**). The notification badge in `DashboardLayout` reads the static `mockNotifications` import directly, so "mark all as read" on the notifications page doesn't decrement the header count (**confirmed**).

## High-level view

The authorization boundary is the most consequential gap. An existing `AdminRoute` component enforces role checks via a `useAdmin` hook, but the new routes were deliberately wrapped in `PrivateRoute` instead. This means the entire clinic management surface (financials, staff PII, settings) is accessible to any authenticated user — patients included.

The data layer is a static in-memory export with no shared state management. Each page imports arrays directly from `mockData.ts`, and mutations happen in component-local `useState` calls. This creates visible inconsistencies: the `DashboardLayout` header badge will never reflect reads made on the notifications page because they operate on independent copies of the data.

`SidebarContent` is declared as a nested component inside `DashboardLayout`'s render body, which means it remounts on every parent re-render and will reset internal state if any is added later.

Invoice creation performs no validation beyond what HTML attributes provide — you can submit an invoice with zero total, negative discount, or blank service names.

The calendar view in Appointments hardcodes January 2024 and renders a fixed 31-day grid. Months with fewer days or different start days will render incorrectly if the mock data or navigation is extended.

## Authorization downgrade from AdminRoute to PrivateRoute

The project already has an `AdminRoute` component that calls `useAdmin()` to verify the current user has an admin role before rendering children. The new routes in `App.tsx` wrap everything under `/admin/*` in `PrivateRoute` instead — a component that only checks `user` existence from `AuthContext`. The diff shows this was a deliberate choice (the FEAT-002 spec mentions "AdminRoute (or PrivateRoute for now)"), but the result is that any authenticated user — including patients who log in to book appointments — can navigate directly to `/admin/billing`, `/admin/staff`, or `/admin/settings` and see financial data, staff PII (emails, phone numbers), and clinic configuration.

```tsx
// Current (from the diff):
<Route path="/admin/billing" element={
  <PrivateRoute>
    <BillingPage />
  </PrivateRoute>
} />

// Expected:
<Route path="/admin/billing" element={
  <AdminRoute>
    <BillingPage />
  </AdminRoute>
} />
```

This is a one-line-per-route fix with zero behavioral risk since `AdminRoute` already exists and is used elsewhere in the codebase.

## Notification state isolation

`DashboardLayout` computes `unreadCount` on every render:

```tsx
const unreadCount = mockNotifications.filter((n) => !n.read).length;
```

This reads the original exported array from `mockData.ts`. Meanwhile, `NotificationsPage` copies that array into local state and mutates the copies when the user clicks "mark as read." The two never communicate. The header badge is frozen at the initial unread count (5), regardless of user interaction on the notifications page.

For a mock-data prototype, the simplest fix is a shared React context that both `DashboardLayout` and `NotificationsPage` consume.

## CreateInvoice validation gaps

The form uses basic HTML form submission with `onSubmit` but has no validation logic:

- No enforcement on the patient `Select` — Radix Select with an empty `value=""` doesn't prevent form submission.
- Line items can have empty `name`, zero `quantity`, and zero `unitPrice`. The subtotal computes to $0.00 and the invoice is "created" successfully.
- The discount field accepts negative values (`parseFloat || 0` normalizes NaN but not negatives). A negative discount increases the total — functionally a surcharge labeled "Discount."

The FEAT-002 spec mentions react-hook-form but the implementation uses raw `useState` with no schema validation layer.

## SidebarContent as a nested component

```tsx
const DashboardLayout = ({ children, title }: DashboardLayoutProps) => {
  // ...
  const SidebarContent = ({ mobile = false }: { mobile?: boolean }) => (
    <div className="flex flex-col h-full">...</div>
  );
  // ...
};
```

React treats this as a new component type on every render because the function reference changes. If `SidebarContent` ever gains internal state (search input, scroll position), that state will be lost on every re-render of the parent. Currently stateless, so the practical impact is limited to unnecessary DOM reconciliation — but it's a latent bug if the component evolves.

## Hardcoded calendar grid

```tsx
const daysInMonth = Array.from({ length: 31 }, (_, i) => i + 1);
// ...
<div /> {/* single empty cell for "January 2024 starting on Monday" */}
{daysInMonth.map((day) => ...)}
```

January 2024 starts on Monday, so 1 leading empty cell is correct for that specific month. The component has no month navigation and the 31-day assumption is baked in. If future work adds month navigation without rewriting this, February will render days 29-31 as invalid dates. The calendar title is also hardcoded rather than derived from `selectedDate`.
